# 资源配置优化（Right Size）

## 功能概述

资源配置优化分析每台虚拟机在观测周期内的真实资源消耗，与当前分配的 CPU 核数和内存容量进行对比，给出"缩减"或"扩容"的具体建议。目的是消除资源浪费（虚配过高）或消除性能风险（虚配不足）。

---

## 工作流程

```
采集历史指标
    ↓
转换为使用率百分比
    ↓
计算统计特征（P95 / P90 / 峰值 / 均值）
    ↓
推算合理的资源需求量
    ↓
归整到标准规格
    ↓
判断调整类型 + 计算浪费比例
    ↓
评估风险等级 + 置信度
    ↓
输出建议
```

---

## 核心算法说明

### 第一步：指标转换

系统从数据库中取出 CPU 和内存的历史指标，统一转换为**使用率百分比**（0–100%）再进行分析，屏蔽不同主机频率、不同内存容量带来的量纲差异。

### 第二步：统计特征计算

对观测周期内每小时一个的采样点，计算以下四个统计量：

| 指标 | 含义 |
|------|------|
| **P95** | 95% 的时间里使用率不超过此值，代表"正常高位" |
| **P90** | 90% 的时间里使用率不超过此值，辅助验证 P95 的稳定性 |
| **峰值（Max）** | 整个观测期内出现的最高使用率 |
| **均值（Avg）** | 平均使用率，反映日常负载水平 |

> P95 和 P90 均使用线性插值计算，比截断取整更精确。

### 第三步：推荐配置算法（P95 + 峰值双重保障）

推荐配置从两个方向同时保护，取较大值作为最终目标使用率：

- **基准需求**：取 P95 和 P90 中的较大值，乘以缓冲系数（预留安全余量）
- **峰值保护下限**：取历史峰值的 80%，确保偶发高负载不会被完全抛弃

```
目标使用率 = max(P95 或 P90 中较大值 × 缓冲系数, 峰值 × 80%)
推荐资源量 = 当前资源量 × 目标使用率 / 100%
```

缓冲系数由评估模式决定（见下方模式说明），默认节省模式为 **1.20**（即预留 20% 余量）。

推荐量计算出来后，会向上归整到最近的**标准规格**：
- CPU：从 `[1, 2, 4, 8, 12, 16, 24, 32, 48, 64]` 核中取最小满足值
- 内存：从 `[0.5, 1, 2, 3, 4, 6, 8, 12, 16, 24, 32, 48, 64, 96, 128, 192, 256]` GB 中取最小满足值
- 最低保证：CPU ≥ 1 核，内存 ≥ 0.5 GB

### 第四步：调整类型判断

CPU 和内存分别独立判断调整方向，各自返回 `cpuAdjustmentType` 和 `memAdjustmentType`：

| 调整类型 | 触发条件 |
|---------|---------|
| **大幅缩减（down_significant）** | 推荐量 ≤ 当前的 50% |
| **缩减（down）** | 推荐量 ≤ 当前的 75% |
| **扩容（up）** | 推荐量 ≥ 当前的 125% |
| **大幅扩容（up_significant）** | 推荐量 ≥ 当前的 150% |
| **配置合理（none）** | 以上条件均不满足 |

### 第五步：浪费比例（wasteRatio）

CPU 和内存分别计算浪费比例 `cpuWasteRatio` 和 `memWasteRatio`，正值代表浪费，负值代表欠配：

```
cpuWasteRatio = (当前CPU - 推荐CPU) / 当前CPU × 100
memWasteRatio = (当前内存 - 推荐内存) / 当前内存 × 100
```

### 第六步：风险等级

通过 CPU 使用率的**变异系数（CV = 标准差 / 均值）**衡量负载波动程度。波动越大，说明该 VM 的使用规律越不稳定，缩减时风险越高：

| 风险等级 | 触发条件 |
|---------|---------|
| **高（high）** | CV > 0.4，负载高度不规律 |
| **中（medium）** | 0.2 < CV ≤ 0.4，有一定波动 |
| **低（low）** | CV ≤ 0.2，负载平稳规律 |

风险为"高"时，建议文字中会追加提示，提醒谨慎操作。

### 第七步：置信度

置信度反映分析结论的可信程度，综合两个因素：

1. **数据充足性**：实际采样点数 / 预期采样点数（观测天数 × 24 小时 × CPU+内存两类指标），上限 100%
2. **数据质量修正**：若 CV 较高（负载波动大），置信度向下修正，质量系数最低降至 0.7

置信度低于阈值（默认 60%）的 VM 不会出现在结果列表中，避免数据不足时给出误导性建议。

---

## 评估模式与参数

不同模式通过调整缓冲系数和置信度阈值，控制建议的保守程度：

| 参数 | 安全模式 | 节省模式（默认） | 激进模式 |
|------|---------|----------------|---------|
| CPU 缓冲系数 | 1.30（+30%） | 1.20（+20%） | 1.10（+10%） |
| 内存缓冲系数 | 1.30（+30%） | 1.20（+20%） | 1.10（+10%） |
| 最低置信度 | 70% | 60% | 50% |

- **安全模式**：缓冲余量大，不容易触发缩减建议，适合对稳定性要求高的生产环境
- **节省模式**：均衡取值，推荐日常使用
- **激进模式**：缓冲余量小，会发现更多优化机会，建议在验证环境中参考使用

## 如何看结果

### 结果字段含义

| 字段 | 含义 | 参考建议 |
|------|------|---------|
| `currentCpu` / `currentMemoryGb` | VM 当前分配的 CPU 核数 / 内存 GB | — |
| `recommendedCpu` / `recommendedMemoryGb` | 算法推荐的配置 | 调整目标 |
| `cpuP95` / `memoryP95` | 95% 时间内的使用率上限 | 最重要的参考值，反映正常高负载水平 |
| `cpuP90` | 90% 时间内的使用率上限 | 辅助验证 P95 的合理性 |
| `cpuMax` / `memoryMax` | 整个观测期的峰值 | 了解极端情况，谨慎参考 |
| `cpuAvg` / `memoryAvg` | 平均使用率 | 日常负载水平参考 |
| `cpuAdjustmentType` / `memAdjustmentType` | CPU / 内存各自的调整类型 | 操作方向 |
| `cpuWasteRatio` / `memWasteRatio` | CPU / 内存各自的浪费比例，正值=浪费，负值=欠配 | 优先处理高正值的 VM |
| `mismatchType` | CPU/内存配比错配类型 | 识别资源分配不均衡的 VM |
| `riskLevel` | 调整风险（高 / 中 / 低） | 高风险时先观察再操作 |
| `confidence` | 置信度（0–100%） | 低于 70% 的结论需谨慎参考 |
| `reason` | 具体数据依据（文本摘要） | 核查建议是否合理的依据 |

### 典型场景解读

**场景一：明显过度配置（最常见）**
- P95 均在 10% 以下，均值 3–5%，峰值不超过 30%
- cpuAdjustmentType / memAdjustmentType = `down_significant`，wasteRatio 较大正值
- **操作建议**：可放心按推荐配置缩减，优先处理风险等级为"低"的 VM

**场景二：使用率波动大**
- P95 为 60%，但 Max 达到 95%，CV 较高，风险等级为"高"
- **操作建议**：先排查峰值产生原因（定时任务？批处理？），再决定是否调整

**场景三：配置不足（需扩容）**
- P95 超过 85%，cpuAdjustmentType / memAdjustmentType = `up` 或 `up_significant`
- **操作建议**：应尽快扩容，避免性能瓶颈影响业务

**场景四：置信度偏低**
- confidence < 70%，通常因为采集天数不足（少于 7 天）
- **操作建议**：延长采集周期后重新分析，当前结论仅供参考

## 标准规格说明

推荐配置会归整到以下标准规格（向上取整），目的是便于实际操作和资源规划：

- **CPU**：1、2、4、8、12、16、24、32、48、64 核
- **内存**：0.5、1、2、3、4、6、8、12、16、24、32、48、64、96、128、192、256 GB

例如，算法算出需要 5.3 核 CPU，实际推荐为 **8 核**（取下一个标准值）。算法算出需要 4.8 GB 内存，实际推荐为 **6 GB**（取下一个标准值）。

---

## 注意事项

1. **分析基于历史数据**：结果反映观测周期内的负载规律，若业务即将发生重大变化（迁移、扩量），应结合业务计划判断
2. **缩减操作不可逆风险低，扩容有资源限制**：缩减后如发现性能不足可随时回调；扩容前需确认宿主机有足够剩余资源
3. **峰值不等于正常使用率**：Max 字段反映极端情况，通常由偶发任务触发，不应作为配置上限的唯一参考
4. **高风险不等于不能操作**：风险等级高只是提示负载不规律，若 P95 仍明显低于当前配置，同样可以适度缩减
