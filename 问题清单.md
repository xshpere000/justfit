## Bug 1 — rightsize 分析未过滤关机 VM（根本原因）

**文件**: `backend/app/analyzers/rightsize.py`
**位置**: `RightSizeAnalyzer.analyze()` 第 60 行
**问题**:
`for vm_id, metrics in vm_metrics.items():` 遍历全部有指标的 VM，不区分开机/关机状态。
关机 VM 的历史指标是关机前的数据，对它做 rightsize 推荐：
- 无实际意义（VM 已关机无法热调整）
- 使 `freed_cpu_from_resource` 计入了关机 VM 的配置差量，虚增释放量
- 导致后续 `needed_cpu = vm_running_cpu - total_freed_cpu` 偏小，负载率虚低
- 可能让本不该释放的主机出现在可释放列表中

**修复方案**: 在 `_analyze_vm` 开头检查 `vm_info.get("power_state")`, 若为关机状态直接跳过。

## Bug 2 — 健康分析 VM 关联逻辑错误（VM 始终无法关联到集群）

**文件**: `backend/app/analyzers/health.py`
**位置**: `_build_cluster_data()` 第 195 行
**问题**:
```python
vm_cluster_key = f"{vm.datacenter}:{vm.datacenter}"  # BUG: 两个都是 datacenter!
```
应为 `f"{vm.datacenter}:{vm.cluster_name}"` 或直接用 `vm.datacenter`。
由于 VM 没有 `cluster_name` 字段（见 `models/resource.py`，VM 只有 `datacenter`），
当前主键是 `datacenter:datacenter`，永远无法与 `datacenter:cluster_name` 匹配，
导致所有 VM 全部走 fallback 的 `host_name in cluster_name` 模糊匹配，
匹配逻辑也不正确（`cluster_info["cluster"].name in vm.host_name` 通常为 False）。
**实际效果**: 超配率计算时 `allocated_cpu` 永远为 0，超配得分虚高（100分），健康分虚高。

**修复方案**: VM 无 cluster_name 字段，应改为通过 Host 关联——`VM.host_name` → `Host.name` → `Host.cluster_name` → `Cluster.name`。

## Bug 3 — idle 去重依赖 vmId，但 vmId 可能无法从 findings 读回

**文件**: `backend/app/services/analysis.py`
**位置**: `calculate_host_freeability()` 第 1037 行 & `_save_findings()` 第 714 行
**问题**:
`_save_findings` 把除 `vmName/confidence/severity/title/description/recommendation/reason` 以外的字段存到 `details` JSON。
`vmId` 确实会存入 `details`（idle_detector 输出中包含它）。
但 `_detect_powered_off_zombie` 返回的字段名是 `"vmId": vm_id`，
而 `_detect_idle_powered_on` 同样返回 `"vmId": vm_id`——这两处都有，本逻辑基本正确。
**真正的问题**: 仅当 idle_vm_ids 成功收集时才能去重，如果 idle finding 的 `details` 解析失败（如 None），会让对应 VM 被重复计入两次（idle 释放 + rightsize 释放）。当前逻辑已有 `except Exception: pass`，但无法区分「vmId 字段不存在」与「JSON 解析失败」。
实际影响较小，但应改为记录警告。

**修复方案**: 加 `if vm_id is None: logger.warning(...)` 并对 vmName 做兜底 key。

## Bug 4 — freed_cpu_pct / freed_mem_pct 分母错误

**文件**: `backend/app/services/analysis.py`
**位置**: 第 1185-1186 行
**问题**:
```python
freed_cpu_pct = round(total_freed_cpu / vm_total_cpu * 100, 1)    # vm_total_cpu 包含关机VM
freed_mem_pct = round(total_freed_memory / vm_total_mem_gb * 100, 1)  # 同上
```
`vm_total_cpu` 是全部 VM（包含关机）的配置总核数。
`total_freed_cpu` 中来自 rightsize 的部分已按 Bug 1 修复后仅针对开机 VM。
来自 idle 的开机闲置 VM 部分是从 running 中释放，但分母却用 total。
百分比偏低，显示不准确（如实际释放了 30%，但显示 20%）。

**修复方案**: 改为 `vm_running_cpu` 和 `vm_running_mem_gb` 作为分母（或使用主机物理总量作为分母更合理）。

## Bug 5 — 主机 VM 映射 key 与主机 key 可能不一致

**文件**: `backend/app/services/analysis.py`
**位置**: 第 1121 行 vs 第 1138 行
**问题**:
VM 建映射时:
```python
key = vm.host_name or vm.host_ip or ""
```
主机建 key 时:
```python
host_key = host.name or host.ip_address or ""
```
vCenter: VM 存 `host_name`（主机 FQDN），Host 存 `name`（同为 FQDN）→ 通常匹配
UIS: VM 存 `host_ip`，Host 存 `ip_address` → 通常匹配
**但边界情况**: 若 VM 的 `host_name` 非空但与 Host 的 `name` 大小写或格式不同，或 vCenter VM 只存了 `host_ip` 而 Host 只存了 `name`，则 key 不匹配，`vm_count=0`，主机被 `continue` 跳过，永远无法进入可释放列表。

**修复方案**: 建立双向映射（name+ip 都建索引），VM 的 host_name/host_ip 两个字段都尝试匹配。

## Bug 6 — `_build_optimization_recommendation` 中 `needed_cpu` 计算逻辑偏差

**文件**: `backend/app/services/analysis.py`
**位置**: 第 1315-1316 行
**问题**:
```python
needed_cpu = vm_running_cpu - total_freed_cpu
needed_mem = vm_running_mem_gb - total_freed_mem
```
`vm_running_cpu` = 全部开机 VM 的**配置**总核数（不是实际使用量），
用配置量减去优化释放量，得到的 `needed_cpu` 是"优化后开机 VM 配置总核数"，
再除以 `retained_cpu`（保留主机物理核数），得到的是"配置超配比"而非"实际负载率"。
如果超配比为 1.5，即使真实使用率只有 30%，这里会显示 CPU 负载 150%（甚至被钳为 0 后显示 0%），结论严重失真。

**修复方案**: 此问题根本上需要实际使用率数据才能算准，当前无法获取。
**折中方案**: 加上文字说明"基于配置量估算"，并合理设置一个默认平均使用率系数（如 30%）来修正，或直接改用"配置超配率"而非"负载率"的语义呈现。

## Bug 7 — `needed_cpu < 0` 被钳为 0 导致 cpu_load_pct 显示 0% 掩盖问题

**文件**: `backend/app/services/analysis.py`
**位置**: 第 1318-1323 行
**问题**:
当 Bug 1 修复前，rightsize 释放量包含关机 VM，导致 `total_freed_cpu > vm_running_cpu`，
`needed_cpu` 为负，被钳为 0，`cpu_load_pct = 0`，显示健康绿色，完全错误。
即使 Bug 1 修复后，也存在计算语义问题（见 Bug 6）。

**修复方案**: Bug 1 修复后此问题基本消除，但需要加断言日志，当 needed_cpu < 0 时打警告日志。

## Bug 8 — 健康分析负载均衡按 datacenter 分组主机，应按 cluster_name 分组

**文件**: `backend/app/analyzers/health.py`
**位置**: `_analyze_balance()` 第 370 行
**问题**:
```python
cluster_hosts = [h for h in hosts if h.datacenter == cluster.datacenter]
```
用 `datacenter` 分组主机，但一个 datacenter 下可以有多个 cluster。
同一 datacenter 所有主机被当作一个集群来计算 CV，导致负载均衡评分不准。

**修复方案**: 改为 `h.cluster_name == cluster.name`（Host 有 `cluster_name` 字段）。
